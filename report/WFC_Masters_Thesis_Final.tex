\begin{abstract}
This report presents a streaming extension of Wave Function Collapse (WFC) for chunked terrain generation under bounded memory and deterministic revisit recovery. Classical WFC is designed for bounded offline grids; the system studied here instead extends terrain incrementally, so each new chunk must be solved as a legal extension of an already committed partial world while preserving local adjacency rules, cross-chunk seam consistency, and replay-based recoverability after eviction.

The principal contribution is validation-enforced correctness in a streaming formulation of WFC that combines propagator-based boundary restriction, backtracking-first local solving, strict seam and internal validation before commit, bounded active-memory eviction, and replay-based reconstruction of committed chunks from stored seed records. The report presents the problem formally, justifies the data structures and algorithms used in the implementation, and separates worst-case analysis from measured runtime behaviour.

The evidence base is intentionally narrow and canonical. Controlled experiment bundles show that backtracking outperforms restart on identical seeded $10 \times 10$ and $20 \times 20$ instances, while the canonical four-direction endurance run verifies online generation, bounded-memory operation, and revisit correctness over 1000 chunks. Under the selected live configuration ($10 \times 10$ chunks, $8 \times 8$ viewport, cache limit 12, $h=0$), mean per-generated-chunk time was 2.58 ms, the 95th percentile was 3.38 ms, and the maximum observed latency was 12.11 ms. Scaling stops at $20 \times 20$ because the streaming mean rises to 17.46 ms, above the 16 ms interactivity target.
\end{abstract}

\chapter{Introduction}
Wave Function Collapse is a widely used procedural generation technique because it produces locally coherent outputs from a compact compatibility specification \cite{karth2017wfcwild,karth2022wfc}. However, standard formulations do not directly address the problem studied here: incremental terrain generation in which previously observed regions are already committed.

This report is framed as an algorithms-and-data-structures study rather than as an implementation demonstration. Accordingly, it emphasizes formal problem definition, justification of algorithmic and data-structure choices, complexity analysis, and experimental validation.

The final deliverable is presented as a technical algorithmic study: formal problem definition, explicit data-structure rationale, correctness argument, complexity analysis, and controlled empirical validation tied to canonical evidence files.

\section*{Central Claim}
The central claim of the final report is:
\begin{quote}
A backtracking-first streaming extension of WFC can generate chunked terrain correctly in four directions under bounded memory, provided that new chunks are seeded by propagator-based boundary restriction, committed only after seam and internal validation, and reconstructed deterministically via replay from persistent seed records.
\end{quote}

\chapter{Problem Description and Motivation}
\section{Problem Significance}
Procedural content generation is a central technique in games, simulation, and virtual environments because it enables large worlds without storing every detail explicitly \cite{shaker2016pcg}. WFC is particularly attractive because it produces outputs that preserve local structure from a tileset or example source \cite{gumin2016wfc,karth2022wfc}. Yet standard WFC assumes a bounded grid and allows failure to be handled by restarting from scratch.

That assumption breaks in a streaming world. Once a player has seen terrain, that terrain becomes part of the committed state of the world. A new chunk is not a fresh solve on an empty grid. It is a constrained extension problem.

\section{Algorithmic Challenges}
The project is algorithmically non-trivial for four reasons:
\begin{enumerate}
\item the world is only partially known at any time,
\item boundary cells are constrained by already committed outside neighbors,
\item memory is bounded, so chunks must be evicted,
\item revisit behavior must be deterministic after storage-backed replay.
\end{enumerate}

\chapter{Related Work and Background}
\section{Wave Function Collapse}
The original WFC implementation by Gumin popularized constraint-based content generation from local adjacency rules \cite{gumin2016wfc}. Karth and Smith later analyzed WFC more formally and argued that it is best understood as a constraint-solving procedure rather than a mysterious generative heuristic \cite{karth2017wfcwild,karth2022wfc}. This perspective is especially useful for the present project because the streaming problem is fundamentally a constrained extension problem.

Most prior WFC discussions focus on bounded offline synthesis where the solver can restart globally and no previously observed output must remain immutable. The present work targets a different operational regime: online extension under committed-neighbor constraints, bounded active memory, and deterministic revisit semantics. That shift from ``bounded generation'' to ``incremental extension'' is the key framing distinction used throughout this report.

\section{Procedural Content Generation}
Procedural content generation research covers rule-based, search-based, grammar-based, and constraint-based techniques \cite{shaker2016pcg}. The relevance of that broader literature here is twofold. First, it clarifies that WFC is one member of a larger PCG family rather than an isolated curiosity. Second, it highlights the specific difficulty of interactive systems: a generator must be judged not only by output quality, but also by latency, determinism, and memory behavior.

This distinction matters for evaluation design. A method that is acceptable in offline PCG may still be unsuitable for interactive streaming if it produces unstable latency tails, rewrite-on-revisit behavior, or uncontrolled memory growth. The final experimental protocol therefore emphasizes online-system metrics in addition to local-solver statistics.

\section{Constraint Satisfaction and Propagation}
The final project relies on standard CSP concepts. Arc consistency and propagation are classical tools for removing values that can no longer be supported by neighboring domains \cite{mackworth1977consistency,dechter2003constraint}. Backtracking search then explores the remaining choices systematically. The project does not claim a new CSP theory result. Its contribution is the adaptation of these ideas into a streaming WFC setting with committed-neighbor constraints and storage-backed revisit semantics.

In particular, the report distinguishes algorithmic novelty from systems-integration novelty. The contribution is not a new asymptotic bound for CSP search. Instead, it is the integration of propagation, search, validation, and persistence into one correctness-preserving online pipeline, then evaluated under controlled and endurance workloads.

\section{Established Foundations and Project Contribution}
The final system is built on several established ideas from the procedural-generation and CSP literature \cite{gumin2016wfc,karth2017wfcwild,karth2022wfc,mackworth1977consistency,dechter2003constraint,cormen2009clrs}. In particular, the following components are standard:
\begin{itemize}
\item WFC as a constraint-based generation method,
\item arc-consistency style propagation,
\item chronological backtracking,
\item asymptotic reasoning for search and storage.
\end{itemize}

The project-specific contribution is not a new foundational solver theory. Rather, it is the integration of these established components into a streaming setting with committed-neighbour constraints and bounded memory. Concretely, the report contributes the following:
\begin{itemize}
\item incremental CSP framing for chunk streaming,
\item propagator-based boundary seeding under committed neighbors,
\item strict pre-commit seam/internal validation,
\item bounded-memory chunk eviction with deterministic revisit recovery,
\item controlled comparison between restart and backtracking within the same problem family.
\end{itemize}

\section{Interim Findings}
The progress report represented the midpoint of the project and captured two key discoveries. First, naive direct boundary assignment is not a valid way to extend a committed WFC world because it can create arc-inconsistent corner states before search even begins. Second, introducing a larger local context around the target chunk appeared promising as a way to reduce boundary pressure. The progress report therefore proposed a design centred on propagator-based seeding, halo-based local solving, and a hybrid contradiction-recovery policy combining restart and backtracking.

The progress report also framed the problem correctly as an online incremental CSP rather than as repeated independent bounded solves. That conceptual framing remains a central continuity across the semester and is preserved in the final document.

\section{Design Refinement}
The final report differs from the interim design in several substantive ways. These revisions follow directly from controlled experiments and full-system validation.

\paragraph{Halo.} The progress report treated halo-based solving as the most promising conditioning strategy. The final report retains halo as an evaluated design option, documents it formally, and reports the corresponding ablation results. However, the measured benchmark evidence does not support halo as the preferred final configuration, so the selected live setting uses $h=0$.

\paragraph{Solver policy.} The progress report emphasised a hybrid online solver. The final report narrows this claim. Backtracking is the primary algorithmic method because it is the more principled online extension strategy and because it outperforms restart in the controlled comparison. Restart is retained only as a bounded live fallback and as a comparison baseline.

\paragraph{Input pipeline.} The final system uses the XML-defined Summer tileset as a fixed and reproducible rule source. This is a deliberate scoping decision: using a fixed tileset keeps the experimental family controlled and allows the report to focus on streaming algorithmics rather than on rule-extraction variability.

\paragraph{Evidence policy.} The progress report described several preliminary experiment families. The final report consolidates those into two canonical evidence sources so that every numeric claim can be traced directly to a stable result file.

\section{Final Scope and Claim}
The final report therefore adopts a narrower but stronger claim than the earlier project documents. It does not claim a new general-purpose WFC theory, and it does not claim that every evaluated design hypothesis was successful. Instead, it claims that the following integrated system has been designed, analysed, and validated:

A validation-enforced streaming extension of WFC based on propagator-seeded incremental solving, strict pre-commit validation, bounded active memory, and deterministic revisit recovery.

\chapter{Formal Problem Formulation}
\section{World and Tile Model}
Let $D$ be the finite set of tile types and let $R_d \subseteq D \times D$ denote the compatibility relation for direction $d \in \{\text{right}, \text{down}, \text{left}, \text{up}\}$. The world is a grid over integer coordinates:
\[
W : \mathbb{Z}^2 \to D,
\]

The world is partitioned into square chunks of size $G \times G$. A chunk is identified by integer chunk coordinates $(c_x, c_y)$ and contains tile assignments for the corresponding $G^2$ world cells.

\section{Committed World State}
Let $K \subset \mathbb{Z}^2$ be the set of committed cells. These are cells that have already been generated and either remain in active memory or can be recovered from persistent storage. Once a cell enters $K$, its assignment is immutable for correctness purposes.

\section{Incremental Extension Problem}
When a target chunk $Q$ is requested, the system must compute a chunk assignment $A_Q$ such that:
\begin{enumerate}
\item every adjacency inside $Q$ satisfies the compatibility relations,
\item every adjacency between a cell in $Q$ and an already committed neighbor in $K$ also satisfies the compatibility relations,
\item the solve does not modify any cell already in $K$.
\end{enumerate}

\section{Correctness Properties}
The final system is evaluated against three properties.

\textbf{Seam correctness.} For any adjacent cells crossing a chunk boundary, the ordered tile pair must belong to the compatibility relation for the relevant direction.

\textbf{Internal correctness.} Every adjacent pair of cells inside a generated chunk must satisfy the same compatibility relation.

\textbf{Revisit consistency.} If a chunk is evicted from memory and later replayed from storage, the replayed chunk must match the previously committed chunk exactly.

\section{Objective}
Given a requested chunk and a committed partial world, compute a legal incremental extension while keeping active memory bounded and maintaining interactive performance for the selected live configuration.

\section{Decision Version of the Incremental Problem}
For algorithm-analysis purposes, it is useful to define the yes/no form of the task.

\textbf{Input.} A committed partial world state $K$, a target chunk coordinate $q$, chunk size $G$, tile alphabet $D$, and directional compatibility relations $R_d$.

\textbf{Question.} Does there exist an assignment for the target local region such that all internal adjacencies are legal, all boundary adjacencies to $K$ are legal, and no cell already in $K$ is modified?

The optimization goal in this project is to answer this decision problem online while minimizing per-chunk latency under bounded active memory. This separation between feasibility and latency objective makes the analysis more precise: correctness is defined by feasibility, while performance is defined by response time and memory behavior.

\chapter{Proposed Algorithm and Data Structures}
\section{Method Definitions}
\textbf{Direct boundary seeding} forces a boundary cell to equal a committed neighbor's tile. This is invalid at corners because the same cell may be forced by multiple outside neighbors.

\textbf{Propagator-based boundary seeding} intersects the set of tiles allowed by each committed outside neighbor. This preserves the CSP interpretation of WFC as domain reduction.

\textbf{Halo-based solving} solves an enlarged $(G+2h)\times(G+2h)$ region and commits only the central $G\times G$ core.

\textbf{No halo} is the special case $h=0$. In the selected live configuration, this is the final choice.

\textbf{Backtracking WFC} uses undo-based DFS over unresolved cells.

\textbf{Restart WFC} discards the current partial solve and tries again from scratch.

\section{Limitations of Baseline WFC for Streaming}
Classical WFC assumes a bounded grid and commonly handles contradiction by restarting from scratch. In a streaming world, both assumptions become problematic.

\paragraph{Bounded-grid assumption.} A new chunk is not solved in isolation. It is solved adjacent to already committed neighbors whose assignments cannot be changed. The solve therefore begins with non-trivial external constraints rather than with a blank domain product.

\paragraph{Restart assumption.} Restart is acceptable for an offline image where failure only wastes time. In a streaming system, restart has two disadvantages. First, it does not preserve progress inside a hard local instance. Second, repeated restart attempts create unbounded wall-clock latency. These issues matter even if restart is restricted to the current local region rather than to the whole world.

These observations motivate the final online design: boundary constraints are integrated through domain restriction, candidate regions are solved under committed-neighbor context, and only validated regions are allowed to enter the committed world.

The implementation couples player motion, chunk scheduling, local solving, persistent storage, and rendering into a single loop. The point of the architecture is not merely software modularity; it is algorithmic separation of concerns.

Each chunk request passes through the following lifecycle:
\begin{enumerate}
\item determine whether the requested chunk is already present in active memory;
\item if not, check persistent storage for a previously committed version;
\item if no stored replay record exists, build a candidate region around the requested chunk and seed it from committed neighbors;
\item solve the candidate region using the chosen local solver;
\item validate every candidate chunk internally and across all relevant seams;
\item commit validated chunks to active memory and persist their seed records;
\item evict distant chunks once the active-memory cap is exceeded.
\end{enumerate}

This lifecycle matters to correctness because it separates \emph{candidate state} from \emph{committed state}. Candidate chunks can be rejected and discarded. Committed chunks cannot be rewritten, and their stored seed records remain sufficient to replay the chunk exactly on revisit.

\section{Boundary Seeding by Propagator Restriction}
For a cell $x$ adjacent to a committed outside tile $t$ in direction $d$, the domain is restricted by:
\[
D(x) \leftarrow D(x) \cap \{u \in D \mid (u,t)\in R_d\}.
\]
If multiple outside neighbors constrain the same cell, their allowed sets are intersected. This is the core fix for the boundary-seeding problem.

This formulation is the algorithmic turning point of the project. It respects the semantics of WFC as a domain-pruning CSP process. A committed neighbor does not dictate a tile value for the new chunk; it merely excludes values that are incompatible with the committed boundary condition.

\begin{algorithm}[H]
\caption{Boundary Seeding by Propagator Restriction}
\begin{algorithmic}[1]
\Require target region $Q$, committed world $K$, compatibility relations $R_d$
\Ensure seeded domain map for cells in $Q$
\State initialize seeded domains for every cell in $Q$ to the full tile set
\ForAll{cell $x$ on or near the constrained boundary of $Q$}
  \ForAll{directions $d$ with committed neighbor $y \in K$}
    \State $A \gets \{u \in D \mid (u, W(y)) \in R_d\}$
    \State $D(x) \gets D(x) \cap A$
  \EndFor
\EndFor
\State \Return seeded domains
\end{algorithmic}
\end{algorithm}

\paragraph{Loop invariant for Algorithm 1.}
After processing any prefix of constrained neighbor-directions for a boundary cell $x$, the current domain $D(x)$ equals the intersection of all compatibility sets already processed for $x$.

\paragraph{Initialization.}
Before processing constraints, $D(x)$ is initialized to the full tile set, which is exactly the intersection over an empty set of constraints.

\paragraph{Maintenance.}
Each processed committed neighbor contributes one allowed set $A$. The update $D(x) \leftarrow D(x) \cap A$ preserves the invariant by extending the running intersection with exactly one new constraint.

\paragraph{Termination.}
When all relevant neighbors are processed, $D(x)$ is the full intersection of all boundary-compatibility constraints for $x$. Therefore the seeded domain contains only boundary-compatible values.

\section{Backtracking Solver}
The solver applies seeded domain restrictions, selects the minimum-entropy unresolved cell, branches over legal tile choices, propagates bans, and undoes recorded changes on failure.

Two implementation details matter in the streaming setting.

\paragraph{Undo records instead of whole-state copying.} The solver records only the domain and support-count changes caused by a speculative choice. This makes reversal proportional to the amount of local propagation actually triggered by that choice.

\paragraph{Minimum-entropy branching.} The variable-ordering heuristic remains WFC-like: the next cell is chosen by minimum entropy rather than by fixed scan order. In the final implementation, this preserves the connection between classical WFC and complete backtracking search while still allowing formal search-tree analysis.

\begin{algorithm}[H]
\caption{Backtracking WFC with Undo Records}
\begin{algorithmic}[1]
\Require state $S$ with wave domains, support counts, and seeded restrictions
\Ensure solved grid or failure
\State apply seeded restrictions and propagate
\If{contradiction detected} \State \Return failure \EndIf
\State \Return \Call{DFS}{$S$}
\Function{DFS}{$S$}
  \If{all cells are resolved}
    \State \Return success
  \EndIf
  \State $c \gets$ unresolved cell with minimum entropy
  \ForAll{tiles $t$ currently legal at $c$}
    \State $\Delta \gets$ empty change record
    \State collapse $c$ to $t$ and propagate, recording all bans/support updates in $\Delta$
    \If{no contradiction and \Call{DFS}{$S$} succeeds}
      \State \Return success
    \EndIf
    \State undo $\Delta$ in reverse order
  \EndFor
  \State \Return failure
\EndFunction
\end{algorithmic}
\end{algorithm}

\section{Restart as Comparator and Bounded Fallback}
The final report keeps restart for two reasons only.
\begin{enumerate}
\item It provides a meaningful baseline in the controlled comparison because restart is the most common contradiction-handling policy in non-streaming WFC.
\item It serves as a bounded engineering fallback in the live browser surface if a single local solve becomes unusually expensive.
\end{enumerate}

However, the report does not treat restart as the main algorithmic answer to the streaming problem. Restart is attractive on easy instances because it can succeed quickly, but it provides no structural guarantee of progress on hard seeded instances. The controlled experiments therefore compare restart directly against backtracking, while the correctness narrative of the report is built around backtracking-first incremental solving.

It expands the local solve region from $G \times G$ to $(G+2h)\times(G+2h)$, then commits only the central core. The intended purpose is to soften boundary pressure by allowing additional local freedom outside the target chunk.

\section{Streaming Scheduler}
The world manager maintains a visible set, a frontier set, a queue of pending generation regions, active memory, and persistent storage.

\begin{algorithm}[H]
\caption{Streaming Chunk Generation}
\begin{algorithmic}[1]
\Require player position, visible window, committed world, cache limit
\Ensure updated world state after a movement step
\State update player position and movement direction
\State queue any visible chunks that are missing
\State queue frontier chunks when the player approaches a chunk boundary
\While{there exists a queued region}
  \State build seeded constraints from committed neighbors
  \State solve the candidate region with backtracking-first WFC
  \If{the candidate region passes seam and internal validation}
    \State commit chunks to active memory and persistent storage
  \Else
    \State reject the candidate and try the next fallback region
  \EndIf
\EndWhile
\State evict distant nonessential chunks when active memory exceeds the limit
\end{algorithmic}
\end{algorithm}

The scheduler is deliberately simple. Immediate visible holes are always higher priority than speculative frontier regions, because user-facing latency is driven by visible demand. This also explains why flat array-based queues remain sufficient in the current implementation: the queue remains small under the tested viewport and cache policies, so a heavier priority structure would add complexity without a demonstrated benefit.

\section{Revisit Validation Procedure}
\begin{algorithm}[H]
\caption{Sampled Revisit Validation}
\begin{algorithmic}[1]
\Require checkpoint chunk $q$, stored checksum table
\Ensure pass/fail result for sampled revisit
\State record checksum of $q$ at generation time
\State continue exploration until $q$ has been evicted
\State move back to $q$
\State replay $q$ from stored seed record
\State recompute checksum of replayed chunk
\If{replayed checksum equals recorded checksum}
  \State \Return pass
\Else
  \State \Return fail
\EndIf
\end{algorithmic}
\end{algorithm}

\paragraph{Hash-based world access.} The active world is keyed by chunk coordinate. Average-case constant-time retrieval is required because rendering, validation, and scheduling all consult nearby chunks frequently.

\paragraph{Array-based wave state.} A fixed tile alphabet makes arrays preferable to pointer-heavy structures. Domains, support counts, and solver worklists are updated at high frequency, so contiguous array storage keeps the bookkeeping simple and inexpensive.

\paragraph{Persistent storage representation.} Committed chunks are represented in persistent storage by compact seed records and replay metadata rather than by full tile grids. \chapter{Correctness Argument}
\section{Invariant 1: Committed Chunks Are Immutable}
Once a chunk is committed, it is either present in active memory or recoverable from its stored seed record. Generation of a new candidate region never modifies a chunk already in the committed set. Therefore previously observed terrain cannot be rewritten by later solves.

\section{Invariant 2: Only Validated Chunks Are Committed}
Candidate chunks are committed only after:
\begin{enumerate}
\item internal validation against the tile compatibility rules, and
\item seam validation against already committed neighbors and any jointly solved neighbors.
\end{enumerate}
Hence any chunk added to the committed world preserves both internal correctness and seam correctness at commit time.

\section{Invariant 3: Propagator Seeding Preserves Only Compatible Boundary Choices}
Boundary restriction removes tiles that are incompatible with committed outside neighbors. It never forces an arbitrary tile that lacks support. Therefore the seeding step preserves only legal extensions of the committed boundary state.

\section{Invariant 4: Revisit Identity Follows from Storage Persistence}
When a committed chunk is evicted, its seed record and replay metadata are retained in storage. Replay reconstructs the chunk by replaying the deterministic solver from that stored record. This assumes deterministic solver behavior, including fixed tie-breaking and consistent propagation order. Therefore, if the stored record is not corrupted and the replay context is unchanged, the replayed chunk must match the original committed chunk exactly. The sampled checksum-based revisit checks are an empirical validation of this invariant.

\section{Correctness Sketch}
Together, these invariants imply the following reasoning chain:
\begin{enumerate}
\item all committed chunks were validated at commit time,
\item committed chunks are never rewritten,
\item new chunks are seeded only with boundary-compatible domain restrictions,
\item only valid new chunks are committed,
\item evicted chunks are restored by replaying their committed seed records.
\end{enumerate}
Hence the system preserves seam correctness, internal correctness, and revisit identity for the tested runs.

\section{Inductive Interpretation}
The above sketch can be read as an induction on the number of committed chunks.

\paragraph{Base case.} The initial chunk or initial visible region is generated from an unconstrained or lightly constrained starting state and is committed only if it passes internal validation. Therefore the committed world is locally correct at initialization.

\paragraph{Inductive step.} Assume that all chunks committed so far satisfy seam correctness, internal correctness, and immutability. Consider the next candidate region. Propagator-based seeding restricts the candidate only to values compatible with already committed neighbors. The solver searches within that restricted state space. If the candidate fails validation, it is rejected and the committed world is unchanged. If it passes validation, it is committed, and the enlarged committed world still satisfies the same properties. This proves that the properties are preserved across every successful commit.

\paragraph{Revisit argument.} Eviction removes a chunk only from active memory, not from the committed world model. Because storage persists the chunk seed record and replay metadata, replay reconstructs the chunk from that stored state, revisit consistency follows as long as the stored representation is intact and the replay context matches. The checksum-based sampled revisits are therefore an empirical audit of a property already supported by the storage design.

\section{Main Correctness Statement}
The invariants can be summarized as a theorem-style claim.

\paragraph{Theorem (commit-time preservation).} Assume: (i) candidate regions are seeded only by propagator restriction against committed neighbors, (ii) candidate regions are committed only after internal and seam validation, (iii) committed chunks are immutable and replayed from their serialized seed representation, and (iv) solver behavior is deterministic with fixed tie-breaking and consistent propagation order at replay time. Then after any finite sequence of successful commits and evictions/replays, the committed world satisfies seam correctness, internal correctness, and revisit consistency.

\paragraph{Proof sketch.} By induction on the number of successful commits. The base case holds because the initial committed set is validated before commitment. For the inductive step, assume properties hold before processing the next candidate. Seeding restricts the candidate search space to values compatible with already committed boundaries. If the candidate fails validation, it is not committed and the invariant set is unchanged. If it passes validation, the enlarged committed set preserves internal and seam correctness by construction. Immutability prevents later writes from invalidating already committed chunks. Eviction does not alter committed content, and replay reconstructs the chunk from its committed seed record under deterministic tie-breaking and consistent propagation order, giving revisit consistency. Therefore all three properties are preserved after each successful operation.

\section{Limits of the Correctness Argument}
The report does not claim a global proof that every possible tileset, boundary condition, and chunk schedule will be solvable online. That would require a much broader theoretical treatment of incremental WFC feasibility. The claim is narrower and appropriate for the course setting: given the implemented validation policy and the verified experiments, the committed world maintains the stated correctness properties whenever candidate regions are accepted by the system.

\section{Asymptotic Notation and Pseudocode Derivation}
The report uses three complementary notations.

\begin{itemize}
\item \textbf{$O(\cdot)$} gives an upper bound on worst-case cost.
\item \textbf{$\Omega(\cdot)$} gives a lower bound on unavoidable work.
\item \textbf{$\Theta(\cdot)$} is used when upper and lower bounds match in the same regime.
\item \textbf{$\Sigma$} is used only for run-total aggregation across chunks.
Case-wise per-chunk comparisons are reported as best, average, and worst curves.
\end{itemize}

For a single chunk request, the implementation has the following cost decomposition:
\[
T_{\text{chunk}} = T_{\text{seed}} + T_{\text{solve}} + T_{\text{validate}} + T_{\text{commit}} + T_{\text{persist}}.
\]

From the pseudocode and the implementation structure:
\[
T_{\text{seed}} = O(G^2 |D|), \qquad T_{\text{validate}} = O(G^2), \qquad T_{\text{commit}} = O(G^2), \qquad T_{\text{persist}} = O(G^2).
\]

The solver term is the dominant one. The backtracking pseudocode branches on a cell choice and recurses on a smaller unresolved state, so its recurrence is:
\[
T_{\text{solve}}(n) \le b\,T_{\text{solve}}(n-1) + P(n), \qquad b \le |D|.
\]

\paragraph{Step-by-step recurrence derivation.}
Unrolling the recurrence for $k$ levels gives
\[
T(n) \le b^k T(n-k) + \sum_{i=0}^{k-1} b^i P(n-i).
\]
Setting $k=n$ yields
\[
T(n) \le b^n T(0) + \sum_{i=0}^{n-1} b^i P(n-i).
\]
If $P(\cdot)$ is polynomially bounded, the geometric branching term dominates the sum asymptotically, so
\[
T(n) = O(b^n) \subseteq O(|D|^n).
\]

Therefore the solver has the conservative worst-case bound:
\[
T_{\text{solve}}(n) = O(|D|^n),
\]

while the unavoidable scan-and-validate work gives a lower bound:
\[
T_{\text{chunk}} = \Omega(G^2).
\]

In the fast path measured by the project, the fixed tileset and bounded viewport make the non-search work dominate enough to observe an operationally tight practical regime, so the live configuration empirically approximates
\[
T_{\text{chunk}} = \Theta(G^2)
\]
in the evaluated regime for successfully solved, low-backtrack chunks. This does not remove worst-case behavior; it indicates that the evaluated live setting remains in a low-order regime frequently enough to support interaction.

For a complete run of $N$ generated chunks, the total cost is the sigma-sum
\[
T_{\text{run}}(N) = \sum_{i=1}^{N} T_{\text{chunk},i}.
\]
With bounded active memory and a fixed tileset, the non-search part of this sum scales as $\Theta(NG^2)$, while the empirical variation comes from the solver term.

Measured runs are consistent with this interpretation. On the controlled 10x10 benchmark, backtracking averaged 1.45 ms and restart averaged 3.22 ms. On 20x20, backtracking averaged 11.78 ms and restart averaged 40.37 ms. In streaming mode, 10x10 averaged 3.20 ms per chunk, while 20x20 rose to 17.46 ms and crossed the 16 ms interactivity threshold. These measurements support the expected growth trend and justify the stop rule used for scaling.

\section{Worst-Case Solver Complexity}
The solver is a search procedure, not a divide-and-conquer recurrence. If $n$ unresolved cells remain and the effective branching factor is at most $b \le |D|$, then a conservative recurrence is:
\[
T(n) \le b \, T(n-1) + P(n),
\]

For a fixed neighborhood degree and fixed tile alphabet, one propagation step is polynomial in the local state size. Therefore the exponential term in the branching process is the correct asymptotic driver. This matches the practical observation that increasing chunk area creates a qualitative hardness transition even when all surrounding scheduler and storage operations remain cheap.

\subsection{Why the Master Theorem Does Not Apply}
The Master Theorem applies to recurrences of the form:
\[
T(n) = aT(n/b) + f(n).
\]
The backtracking solver does not split the problem into independent subproblems of size $n/b$. Instead, it branches on one cell at a time, reducing the unresolved count by roughly one level per recursive step. Therefore the Master Theorem is not the correct tool for the main solver recurrence. Applying it here would be methodologically incorrect.

\subsection{Recurrence-Tree Reasoning}
The recurrence tree has depth at most $n$. At level $i$, there can be at most $b^i$ nodes. Summing work over levels gives:
\[
T(n) = O\!\left(\sum_{i=0}^{n-1} b^i P(n-i)\right).
\]
If $P(\cdot)$ is polynomially bounded, the exponential branching term dominates, so the solver has an exponential worst-case upper bound:
\[
T(n) = O(b^n).
\]
With $b \le |D|$, this gives the familiar conservative bound $O(|D|^n)$.

This recurrence-tree discussion shows how the search tree grows as one unresolved cell is fixed per level. It also clarifies why exponential growth appears in chunk area rather than in linear chunk dimension: branching depth is tied to the number of unresolved cells.

\section{Operational Interpretation}
The pessimistic bound is still useful because it explains why larger chunk sizes become difficult quickly. However, the measured live behavior is much better because:
\begin{enumerate}
\item seeded boundary restrictions shrink domains before search begins,
\item propagation removes unsupported values aggressively,
\item many chunks solve with almost no backtracking in the tested live configuration.
\end{enumerate}

The controlled measurements make this distinction concrete for the evaluated sizes. At both $10 \times 10$ and $20 \times 20$, the backtracking solver solves every tested instance with lower mean and lower variance than restart. The asymptotic analysis still explains why search can become hard in larger state spaces, while the empirical claim in this report is bounded to the evaluated benchmark family.

\paragraph{How to read the counters.} In the controlled benchmark, \emph{attempts} counts full solver passes. Backtracking usually reports one attempt because the depth-first search stays inside a single solve trajectory. Restart can exceed one attempt because it may discard a whole partial solve and start again from scratch. In the canonical 1000-chunk endurance run, restart-mode solves are zero because backtracking succeeded for every generated chunk and the restart fallback was never triggered.

\section{Space Complexity}
Active memory stores at most $C$ chunks, each with $G^2$ tile assignments:
\[
S_{\text{active}}(G,C) = O(CG^2).
\]

Persistent storage in the implemented system grows with the number of generated chunks as seed records and replay metadata:
\[
S_{\text{persistent}}(N) = O(N).
\]

If a full-grid baseline were used instead, the storage term would be $O(NG^2)$. The seed-only design avoids that cost, at the expense that revisit reconstruction must re-run local solving:
\[
T_{\text{revisit, seed-only}} = O(T_{\text{solve}}).
\]

The solver state itself stores per-cell domains and support counts:
\[
S_{\text{solver}}(n, |D|) = O(n|D|).
\]

In the selected live configuration, the active-memory cap is deliberately small, so the persistent term dominates long-run growth while the active term remains bounded. This separation is algorithmically desirable: persistence scales with explored world size, whereas active memory scales only with the current operational window.

\section{Amortized Online-Cost Interpretation}
The end-to-end online step includes scheduling, local solving, validation, commit, and occasional eviction/replay. Let $L_i$ be the local solver cost for the $i$-th generated chunk (the dominant variable term). Let $E_i$ and $R_i$ be indicators for whether that step triggers an eviction or replay event. Then a practical upper model is:
\[
T_i = O(L_i + G^2 + C\cdot E_i + G^2\cdot R_i).
\]

Over $N$ generated chunks, if eviction policy keeps $\sum_i E_i = O(N)$ and replay count is $\sum_i R_i = O(N)$, the non-solver maintenance overhead is linear in $N$ and the amortized non-solver overhead per generated chunk remains bounded by a constant that depends on $G$ and cache policy. Therefore the empirical latency profile is primarily governed by the distribution of $L_i$, which is exactly why solver-side statistics (mean, p95, variance, tails) are central in Chapters 9 and 10.

The scaling decision is intentionally simple: the controlled benchmark is useful for comparing solvers, but it does not set the deployment gate. The final live gate uses streaming latency, and because the $20\times20$ streaming mean is 17.46 ms, the report stops before $30\times30$ even though controlled backtracking at $20\times20$ remains lower at 11.78 ms.

The practical implication is architectural: persistence growth is expected and acceptable, while active memory remains intentionally capped for interactive operation. The redundant space-growth plot is omitted here because the tabled counts already convey the same linear trend. A dedicated seed-regeneration experiment in Chapter 10 then quantifies the explicit space-time replacement cost when persistent tile grids are replaced by coordinate-seed metadata.

Chunk generation is path-dependent due to streaming order and boundary constraints.

\paragraph{Theorem (stop-rule consistency).}
Under the project deployment criterion ``maintain average per-chunk streaming latency below 16 ms'', the measured scenario table implies that $10\times10$ is admissible and $20\times20$ is inadmissible for final live scaling.

\paragraph{Proof.}
From Table~\ref{tab:streaming_sizes}, $\overline{T}_{10}=3.20\text{ ms}<16$ and $\overline{T}_{20}=17.46\text{ ms}>16$. Therefore the acceptance predicate holds for $10\times10$ and fails for $20\times20$. By the stated stop policy, further up-scaling (including $30\times30$) is not part of the final deployment pass.

The low average backtrack count at $20 \times 20$ does not contradict the higher latency: in streaming mode, end-to-end time includes scheduling, validation, commit, and persistence work, so chunk size can push latency above the target even when search itself remains shallow.

\chapter{Limitations and Future Work}
\section{Limitations}
\begin{itemize}
\item The strongest interactive claim is bounded to the selected $10\times10$ live configuration. At $20\times20$ and $30\times30$, measured latency and variance move outside real-time targets, so those settings are treated as analytical stress regimes rather than deployment regimes.
\item The evaluation uses one fixed tileset family (Summer XML). This controls variability and improves internal comparability, but it limits immediate generalization to tilesets with very different rule density, symmetry, or entropy structure.
\item Revisit consistency is verified by two bounded audits: four storage-replay checkpoints in the canonical endurance run and 100 deterministic seed-replay checkpoints under matched boundary snapshots. This is stronger than the earlier 4/4 sample only, but still not an exhaustive all-chunk proof over arbitrary trajectories.
\item The live browser surface retains bounded restart fallback for robustness. This is an engineering safeguard, but it means the demonstration layer does not represent a pure backtracking-only runtime policy.
\item Performance claims are implementation- and environment-dependent (Node.js runtime, platform, and hardware conditions). The report therefore treats absolute milliseconds as system-specific measurements, not universal constants.
\item Scheduler and storage policies are evaluated as implemented. The report does not claim optimality relative to alternatives such as priority-based queues, predictive prefetching, or richer backing stores.
\end{itemize}

\section{Threats to Validity}
\paragraph{Internal validity.} The report separates canonical evidence from superseded runs and uses deterministic seeds plus checksum-based revisit checks to reduce instrumentation ambiguity. Residual risk remains because instrumentation is embedded in the same implementation being evaluated; independent profiling would further strengthen this axis.

\paragraph{External validity.} Runtime measurements depend on implementation language, runtime environment, and tileset characteristics. The project therefore frames timing as evidence for this system and configuration, while claiming only qualitative transfer (for example, hardness-transition behavior) across other deployments.

\paragraph{Construct validity.} Seam correctness, internal correctness, and revisit consistency align with the formal objectives, but they are not direct measures of terrain aesthetics or player-perceived quality. The report intentionally avoids conflating formal validity with artistic quality.

\section{Future Work}
Several reasonable next steps remain:
\begin{enumerate}
\item evaluate additional tilesets and rule families in order to test how sensitive the final conclusions are to tileset structure and symmetry class;
\item compare the current scheduler against more explicit priority structures such as heaps or frontier-distance ordering;
\item study whether alternative local-region policies can recover a meaningful benefit from halo or from other context-expansion techniques;
\item extend the revisit experiment from sampled checks to larger systematic audits over long trajectories;
\item move persistence from browser-local storage to a richer backing store if the system is scaled beyond course-project scope.
\end{enumerate}

\chapter{Conclusion}
This report presents a final implementation of streaming terrain generation with Wave Function Collapse. The study began as a broad proposal to extend WFC to an unbounded world and converged to a narrower, evidence-bounded contribution: a validation-enforced streaming WFC system formulated as an incremental CSP and evaluated through controlled and long-run experiments.

The main algorithmic conclusion is that \emph{propagator-based boundary restriction} is the correct way to integrate committed neighbours into a new local WFC solve. This refinement resolves the contradiction introduced by direct boundary copying and gives the streaming system a principled boundary-conditioning mechanism.

The main solver conclusion is that \emph{backtracking} is the preferred primary local-search strategy for the selected streaming configuration. For the evaluated tileset and seeded benchmark instances, it outperforms restart in mean and p95 runtime and aligns with the incremental-extension semantics of the problem.

The main systems conclusion is that the final four-direction endurance run demonstrates a functioning online world model rather than an isolated benchmark. The system generates chunks ahead of movement, evicts chunks under bounded memory, replays previously committed chunks from stored seed records, and preserves seam correctness, internal correctness, and sampled revisit identity throughout a 1000-chunk run.

The main practical conclusion is that the selected $10 \times 10$ live configuration achieves the project's interactive target on both mean and p95 chunk-generation latency. The report therefore supports a balanced final claim: small-chunk real-time streaming is achievable in the present system, whereas larger chunk sizes move quickly into the heavy-tail regime predicted by the CSP analysis.

Taken together, these results provide an integrated algorithms study with formal problem definition, explicit method selection, data-structure justification, correctness reasoning, asymptotic analysis, and empirical verification.
